### **F-01.6 Customer Account Profile Management**

> *This feature was implied by the registration and order flows but not explicitly specified in v2.0. Added in v2.1.*

**Summary**

Authenticated customers can view and update their account profile (name, email, phone), change their password, manage saved delivery addresses, and close their account.

**Background / Context**

Without explicit account management APIs, the frontend has no endpoints to render or update the /account/profile and /account/addresses pages referenced throughout the spec. These are foundational self-service capabilities expected by every e-commerce customer.

**User Story**

As a registered customer, I want to update my personal details, manage my saved addresses, and change my password from my account page so that my account reflects current information.

**Flow — Happy Path**

1. Customer navigates to /account/profile. Page loads via GET /api/v1/account/me.
2. Customer updates first name, last name, or phone and clicks Save.
3. PATCH /api/v1/account/me updates only the supplied fields.
4. Customer navigates to /account/addresses. GET /api/v1/account/addresses returns saved addresses.
5. Customer adds a new address (POST), edits an existing one (PATCH), deletes one (DELETE), or sets a default (PATCH with isDefault: true).
6. Customer navigates to /account/security and changes password via PATCH /api/v1/account/password. All existing refresh tokens are revoked (same behaviour as password reset).

**Flow — Edge Cases**

| Edge Case | Expected System Behaviour |
| :---- | :---- |
| Customer attempts to change email | Email change requires re-verification: new email receives verification link. Old email remains active until new one is verified. |
| Customer deletes their default address | If other addresses exist, first remaining address promoted to default. If no addresses remain, defaultAddressId set to null. |
| Customer requests account closure | Initiates soft-delete flow per F-08.2 (Data Erasure). Account deactivated immediately; erasure scheduled per TC-31. |
| Customer sets a non-existent address as default | 404 returned. |

**Business Rules**

| BR \# | Rule |
| :---- | :---- |
| BR-01 | PATCH /account/me accepts any subset of updatable fields (firstName, lastName, phone). Fields not supplied are unchanged. |
| BR-02 | Email change requires re-verification of the new email before it replaces the old one. During the pending window both old and new email addresses are stored. |
| BR-03 | Password change via PATCH /account/password requires the customer to supply their current password for confirmation. |
| BR-04 | Successful password change revokes all refresh tokens (same as F-01.3 reset). |
| BR-05 | Maximum 10 saved delivery addresses per customer. Attempting to add an 11th returns 422. |
| BR-06 | Exactly one address may be flagged as isDefault=true per customer. Setting a new default clears the previous flag atomically. |

**Input Validation Rules**

| Field | Type | Required | Validation Rules | Error Message Shown |
| :---- | :---- | :---- | :---- | :---- |
| firstName | String | No (PATCH) | Min 1, max 50 chars. Letters, hyphens, apostrophes, spaces. Trimmed. | 'First name must be 50 characters or fewer' |
| lastName | String | No (PATCH) | Same as firstName. | 'Last name must be 50 characters or fewer' |
| phone | String | No | E.164 format (e.g. +12025551234). Max 20 chars. Stored after validation. | 'Please enter a valid phone number' |
| currentPassword | String | Yes (for PATCH /password) | Presence check. Validated against stored bcrypt hash before any change is made. | 'Current password is incorrect' |
| newPassword | String | Yes (for PATCH /password) | Same rules as registration (min 8, max 128, ≥1 upper, ≥1 lower, ≥1 number). Must differ from currentPassword. | 'New password must meet complexity requirements' / 'New password must be different from current password' |
| address fields | Various | Yes (for POST /addresses) | line1 (max 100), line2 (optional, max 100), city (max 100), state/county (max 100), postalCode (format per country), country (ISO 3166-1 alpha-2). | Per-field: '[Field] is required' / '[Field] format is invalid' |

**API Contract**

| Method | Endpoint | Description | Request Body / Params | Response |
| :---- | :---- | :---- | :---- | :---- |
| **GET** | /api/v1/account/me | Get profile | *—* | *200: { id, firstName, lastName, email, phone, createdAt }* |
| **PATCH** | /api/v1/account/me | Update profile | *{ firstName?, lastName?, phone? }* | *200: { user }* |
| **PATCH** | /api/v1/account/email | Request email change | *{ newEmail, password }* | *200: { message } / 409: email taken* |
| **POST** | /api/v1/account/email/verify | Confirm change | *{ token }* | *200: { message } / 400: invalid or expired* |
| **PATCH** | /api/v1/account/password | Change password | *{ currentPassword, newPassword }* | *200: { message } / 401: wrong current password* |
| **GET** | /api/v1/account/addresses | List addresses | *—* | *200: { addresses[] }* |
| **POST** | /api/v1/account/addresses | Add address | *{ line1, line2?, city, state, postalCode, country, isDefault? }* | *201: { address } / 422: limit reached* |
| **PATCH** | /api/v1/account/addresses/:addressId | Edit address | *{ line1?, city?, postalCode?, isDefault?, … }* | *200: { address }* |
| **DELETE** | /api/v1/account/addresses/:addressId | Delete address | *—* | *200: { message }* |
| **POST** | /api/v1/account/close | Close account | *{ password }* | *200: { message } — schedules erasure* |
