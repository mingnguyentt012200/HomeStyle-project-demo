## **M05 — Order Management**

Covers the full lifecycle of a placed order: confirmation, history, tracking, and admin fulfilment. Orders are created exclusively via the Stripe webhook (F-04.2) and progress through a defined status machine.

---

### **F-05.3 Admin Order Fulfilment**

**Summary**

Ops admins view the fulfilment queue, update order status, enter tracking numbers, and handle fulfilment exceptions.

**Background / Context**

Orders placed by customers must be physically processed and dispatched. This admin interface is the operational bridge between a confirmed payment and a delivered product.

**User Story**

As an Ops admin, I want to manage the order fulfilment queue — mark orders as processing, enter tracking numbers, and flag exceptions — so that orders are fulfilled accurately and on time.

**Flow — Happy Path**

1. Admin views /admin/orders — filterable by status, date, carrier.
2. Admin selects a confirmed order → changes status to 'processing'.
3. Admin enters carrier name and tracking number → status updated to 'dispatched'. Dispatch email triggered.
4. Carrier delivers → admin (or automated webhook) marks as 'delivered'.

**Flow — Edge Cases**

| Edge Case | Expected System Behaviour |
| :---- | :---- |
| Admin attempts to skip a status (e.g. confirmed → delivered) | Status machine rejects invalid transitions (F-05.4). 422 returned. |
| Tracking number entered incorrectly | Admin can correct via PATCH /admin/orders/:orderId — re-triggers dispatch email only if status changes. |
| Order already cancelled — admin tries to dispatch | 422: 'Cannot dispatch a cancelled order.' |

**Business Rules**

| BR \# | Rule |
| :---- | :---- |
| BR-01 | Status transitions are governed exclusively by F-05.4 Status Machine. No direct DB status overrides from this UI. |
| BR-02 | PATCH /admin/orders/:orderId is the partial-update endpoint for order fields (tracking number, notes, assignedTo). |
| BR-03 | All admin order actions are written to an audit log (adminId, action, timestamp, before/after state). |

**Input Validation Rules**

| Field | Type | Required | Validation Rules | Error Message Shown |
| :---- | :---- | :---- | :---- | :---- |
| status | Enum | Yes (for status update) | Must be a valid next status per F-05.4 state machine for the current status. | 422: 'Invalid status transition from [current] to [requested]' |
| trackingNumber | String | Yes (for dispatch) | Alphanumeric + hyphens. Max 50 chars. Trimmed. | 'Tracking number is required when dispatching' |
| carrier | String | Yes (for dispatch) | Must match a configured carrier slug in admin settings. | 'Please select a valid carrier' |
| adminNotes | String | No | Max 1000 chars. Plain text. | 'Notes must be 1000 characters or fewer' |

**API Contract**

| Method | Endpoint | Description | Request Body / Params | Response |
| :---- | :---- | :---- | :---- | :---- |
| **GET** | /api/v1/admin/orders | List orders | *?status=&dateFrom=&dateTo=&carrier=&page=* | *200: { orders[], total, page }* |
| **GET** | /api/v1/admin/orders/:orderId | Order detail | *—* | *200: { order, lineItems[], statusHistory[], auditLog[] }* |

| **GET** | /api/v1/admin/orders/:orderId/audit | Audit log | *—* | *200: { events[{ adminId, action, timestamp, diff }] }* |
