### F-09.3 Admin Inventory & Stock Management

**Summary**

Admin — Operations users manage the three-layer inventory model — **Physical Stock**, **Reserved Stock**, and **Available to Promise (ATP)** — with explicit Soft Lock / Hard Lock semantics. The runtime catalogue (F-02.3), checkout (F-04.x), routing (F-05.5), and pick/pack (F-05.6) all read these layers to drive PDP availability, reservations, and dispatch SLAs (TC-18).

**Background / Context**

Inventory is a multi-layered state machine, not a single counter (BRD §17.4). The single biggest cause of overselling at peak traffic is treating "stock" as one number. This spec implements the senior-BA approach: ATP is the only number shown to customers (BR-INV-01), Soft Locks (TC-13 Redis SETNX) protect the checkout window, Hard Locks (durable DB rows) protect the paid-but-unshipped window, and Safety Stock absorbs carrier and channel lag (BR-INV-02).

**User Story**

As an Admin — Operations user, I want to view and adjust the three inventory layers (Physical / Reserved / ATP) with audited reasons, manage safety stock and low-stock thresholds, and investigate oversell incidents — so that customers never see stale ATP and dispatch SLAs are reliably met.

---

**Inventory Model (BRD §17.4 / BR-INV-01..10)**

| Layer | Definition | Mutator |
|---|---|---|
| **Physical Stock** | Goods physically on shelves. Ground truth. | F-05.6 pick confirmation (decrement); F-09.3 GRN / cycle count / manual adjustment (increment / decrement). |
| **Reserved Stock** | Active Soft Locks + Hard Locks against this SKU. | F-04.1 checkout init (Soft Lock); F-04.2 payment confirmation (promote to Hard Lock); F-05.6 pick (release Hard Lock); F-05.3 cancellation (release). |
| **ATP — Available to Promise** | `ATP = Physical − Reserved − SafetyStock`. The only number on PDPs. | Derived; recomputed on every Reserved-changing event. |

**Locks**

| Lock | Storage | TTL | Released by |
|---|---|---|---|
| **Soft Lock** | Redis SETNX | TC-13 (15 min) | TTL expiry / cart abandon / payment fail |
| **Hard Lock** | DB row `order_allocations` | None | Pick confirmation / order cancel / Super Admin manual override |

---

**Flow — Happy Path (Stock Adjustment)**

1. Admin opens `/admin/inventory` (SCR-A-10). Default view lists SKUs filtered by low-stock, recent movement, OOS, made-to-order, or oversell-flagged.
2. Admin clicks a SKU → **Inventory Detail**: Physical, Reserved (split: Soft Lock count / Hard Lock count), ATP, Safety Stock, Low-Stock Threshold, recent movements (audit log).
3. Admin chooses **Adjust Stock** → enters delta (signed integer) + mandatory reason (controlled vocab) + optional note.
4. System validates: cannot drive Physical < (current Hard Locked qty) — that would be an immediate oversell.
5. Adjustment commits atomically. `ATP` recomputed. If `ATP` transitions OOS → InStock and a "Notify Me" list exists for this SKU: TC-28 hourly job queued for back-in-stock notification.
6. Adjustment logged in audit (BC-40, BR-INV-08).
7. If ATP crosses Low-Stock Threshold downward: Process 11.7 low-stock alert generated (subject to TC-27: 24-hr cooldown).

**Flow — Happy Path (Safety Stock Review)**

1. Admin (quarterly per **TC-INV-03**) opens **Safety Stock Review** report.
2. Report ranks SKUs by oversell incidents, near-oversell events, carrier-delay return events, and inbound-sync mismatches.
3. Admin updates Safety Stock per SKU (defaults per BR-INV-02: 2 for parcel accessories; 1 for white-glove; 0 for made-to-order).
4. Changes audited; effective immediately at next ATP recomputation.

**Flow — Happy Path (Oversell Incident Triage)**

1. System detects oversell at F-05.6 pick step (Physical < total Hard Locked). Per BR-INV-09: last-paid order auto-moved to `On-Hold`; E-OVERSELL email dispatched.
2. Admin opens `/admin/inventory/oversell-incidents` — list of open incidents.
3. Admin investigates root cause: race condition (system flaw), safety stock too low (config), supplier shortage (operational), cycle-count variance (warehouse).
4. Admin marks incident `Resolved` with root-cause tag. Used to drive monthly Oversell Rate KPI (target ≤0.05%; postmortem if >0.2%).

**Flow — Edge Cases**

| Edge Case | Expected System Behaviour |
|---|---|
| Admin attempts to set Physical < Hard Locked qty | 422 (BR-INV-05): "Cannot reduce Physical below committed Hard Locks (N units). Cancel orders first or contact Super Admin." |
| Cycle count enters negative count | Allowed with `count_correction` reason. System flags SKU for investigation; new ATP recomputed. |
| Admin sets `discontinued` on a SKU with active Hard Locks | Blocked. Requires (a) cancelling those orders or (b) Super Admin override. Soft Locks left to expire naturally. |
| Concurrent admin edit | Optimistic locking via `updated_at`. Second writer 409 with diff preview. |
| Bulk CSV: 5,000 rows, mixed valid / invalid | Per-row validation report; valid rows committed atomically; invalid rows skipped with error column. |
| Customer order cancel happens during admin adjustment | DB transaction serialises; final ATP correct in either order. |
| Channel buffer change (Phase 3 marketplace) | Reduces ATP feed to that channel only; internal ATP unaffected. |

### F-09.3 Extension — Dynamic Safety Stock & Risk-Based Replenishment

*Spec version: 4.0 · Added per BA case study: EU/US inventory intelligence upgrade for 20K+ SKUs*

**Summary**

Replaces the hardcoded ATP ≤ 5 low-stock threshold (applied uniformly across all SKUs) with a **dynamic, configurable safety stock system** driven by per-category and per-SKU configuration. Introduces **risk-based alert prioritisation** (Critical / Medium / Low) based on ATP, sales velocity, lead time, and supplier reliability — so warehouse teams act on the right alerts at the right time.

**Background / Context**

The existing system fires a Low Stock Alert the moment ATP ≤ 5 for any SKU, regardless of category. This produces two simultaneous failure modes:

- **Under-alerting on slow-replenishment items:** A sofa (lead time 28 days, sales velocity 3/day) reaches ATP = 4, giving only 1.3 days of remaining stock. The fixed threshold of 5 fires too late — or is adjusted upward manually and forgotten.
- **Over-alerting on fast-replenishment items:** A table lamp (lead time 2 days, sales velocity 0.1/day) reaches ATP = 4, giving 40 days of remaining stock. The alert is noise; warehouse teams begin ignoring the dashboard.

**Root Cause (BA Analysis)**

Not all low-stock situations carry the same business risk. The system must evaluate ATP *in context* — relative to how fast the SKU sells and how long it takes to replenish. A fixed universal threshold is a governance shortcut, not an inventory intelligence tool.

**ATP Risk Calculation**

```
Days of Stock Remaining = ATP / Sales Velocity (units/day)
```

| Risk Level | Condition | Alert Behaviour |
|---|---|---|
| **Critical** | Days of Stock < Lead Time Days | Real-time alert + dashboard Critical badge |
| **Medium** | Days of Stock < (Lead Time × 1.5) | Dashboard warning (amber) |
| **Low** | Days of Stock ≥ (Lead Time × 1.5) | Silent monitoring only |

Example — Sofa (IT-SOFA-001):
- ATP = 4, Sales Velocity = 3/day → Days of Stock = 1.3 days
- Lead Time = 28 days → 1.3 < 28 → **Critical**

Example — Lamp (LAMP-002):
- ATP = 4, Sales Velocity = 0.1/day → Days of Stock = 40 days
- Lead Time = 2 days → 40 ≥ 3 (2 × 1.5) → **Low** (silent)

**Governance & Approval Workflow**

Finance Manager concern: *"If warehouse staff can set any threshold freely, they will inflate buffers to avoid alerts."*

| Change Size | Approval Required | Who Approves |
|---|---|---|
| < 10% change to safety stock | Auto-approved | — |
| 10%–30% change | Manager approval required | Inventory Manager |
| > 30% change | Finance approval required | Finance Manager |

Approval requests surface in the approver's admin dashboard as a task. Pending requests expire after 5 business days (requester notified). Rejected changes are logged with rejection reason.

**Audit & Versioning**

Every threshold change — approved or auto-approved — creates an immutable audit record:

| Field | Description |
|---|---|
| `old_value` | Safety stock before change |
| `new_value` | Safety stock after change |
| `changed_by` | Admin user ID |
| `approved_by` | Approver ID (null if auto-approved) |
| `change_reason` | Free text (required) |
| `timestamp` | UTC datetime |
| `version` | Config version at time of change |

Historical reports use the `effective_date`/`expiration_date` on the config version active at the time of the report — not the current config.

---

**Updated Alert Logic**

❌ **Old (removed):** `IF ATP <= 5 → trigger Low Stock Alert`

✅ **New:**
```
days_of_stock = ATP / sales_velocity
lead_time     = category_avg_lead_time (or SKU override)

IF days_of_stock < lead_time              → CRITICAL  (real-time alert)
IF days_of_stock < (lead_time × 1.5)     → MEDIUM    (dashboard warning)
ELSE                                      → LOW       (silent monitoring)
```

Additionally: if a previously triggered alert's config changes (safety stock updated), the system recalculates risk level and refreshes the alert status immediately.

---

**Additional Flows**

**Flow — Configure Dynamic Safety Stock (Admin)**

1. Admin navigates to `/admin/inventory/thresholds`.
2. Selects scope: Category or SKU. Chooses warehouse (optional).
3. Enters new `safety_stock`, optional `lead_time_days_override`, optional `sales_velocity_override`, `effective_date`, optional `expiration_date`, and mandatory `change_reason`.
4. System calculates change percentage vs. current active config.
5. If change < 10%: auto-approved, effective from `effective_date`.
6. If 10%–30%: approval request created for Inventory Manager.
7. If > 30%: approval request created for Finance Manager.
8. Pending approval shown in requester's dashboard. Approver sees old vs. new value, change %, reason, and a preview of affected SKU count.
9. On approval: config version written; ATP risk levels recomputed for all affected SKUs immediately.
10. On rejection: original config retained; rejection reason logged.

**Flow — Seasonal Override**

For Christmas collections or promotional events, admin creates a config with `effective_date` = campaign start and `expiration_date` = campaign end. System automatically reverts to the prior config version on expiration.

---

**Edge Cases (Dynamic Safety Stock)**

| Edge Case | Expected System Behaviour |
|---|---|
| Threshold change while existing Critical alerts are active | System re-evaluates risk level of all open alerts immediately after config change. Alerts that no longer qualify as Critical are downgraded; new Critical alerts are raised if previously Low SKUs now qualify. |
| Historical report requests (e.g. last quarter's low-stock events) | Use the config version `effective_date`/`expiration_date` active at the time of the event — not the current config. |
| Supplier shipment delayed (incoming stock not yet received) | In-transit stock does NOT increase ATP (BR-INV-05 existing rule). Risk level reflects current physical reality. If supplier delay is confirmed, admin may manually reduce `lead_time_days_override` expected arrival or raise safety stock temporarily. |
| Sales velocity spike (e.g. viral product) | Velocity is recomputed from rolling 7-day average nightly. Admin can trigger immediate recompute via a "Refresh Velocity" action on the SKU detail page. |
| SKU with no sales history (new product) | Sales velocity defaults to category average. Admin may set `sales_velocity_override` at launch. |
| Config expiration date passes with no successor config | System falls back to the next-priority config in the resolution chain (SKU → Category → System Default). |
| Pending approval request expires (5 biz days) | Request auto-rejected; original config retained; requester and approver notified. |

**SQL Reference — BA Investigation Queries (documented for QA and stakeholder alignment)**

These queries were used during the BA investigation phase to validate the need for dynamic thresholds and to establish baseline category averages. They serve as a reference for developers implementing the velocity and lead-time computation jobs.

```sql
-- Lead Time Analysis (per category)
SELECT
    product_category,
    AVG(DATEDIFF(day, po_created_date, received_date)) AS avg_lead_time
FROM purchase_order_receipts
GROUP BY product_category;
-- Purpose: identify which categories replenish slowly vs. quickly

-- Sales Velocity Analysis (per SKU)
SELECT
    sku,
    AVG(quantity_sold_per_day) AS avg_daily_sales
FROM sales_velocity_daily
GROUP BY sku;
-- Purpose: identify how quickly each SKU sells; drives days_of_stock calculation

-- Risk Classification Preview (for BA validation with stakeholders)
SELECT
    s.sku,
    s.atp,
    v.avg_daily_sales,
    lt.avg_lead_time,
    ROUND(s.atp / NULLIF(v.avg_daily_sales, 0), 1) AS days_of_stock,
    CASE
        WHEN (s.atp / NULLIF(v.avg_daily_sales, 0)) < lt.avg_lead_time
            THEN 'CRITICAL'
        WHEN (s.atp / NULLIF(v.avg_daily_sales, 0)) < (lt.avg_lead_time * 1.5)
            THEN 'MEDIUM'
        ELSE 'LOW'
    END AS risk_level
FROM sku_stock s
JOIN sales_velocity_daily v ON v.sku = s.sku
JOIN (
    SELECT p.category, AVG(DATEDIFF(day, po_created_date, received_date)) AS avg_lead_time
    FROM purchase_order_receipts por
    JOIN products p ON p.sku = por.sku
    GROUP BY p.category
) lt ON lt.category = s.category;
```
