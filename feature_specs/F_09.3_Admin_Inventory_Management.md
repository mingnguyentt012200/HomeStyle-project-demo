## M09 — Admin Operations (Logistics & Commerce Configuration)

---

### F-09.3 Admin Inventory & Stock Management

**Summary**

Ops admins (and Super Admin) view and adjust stock levels, low-stock thresholds, lead times per product configuration, and inventory states (in-stock, made-to-order, pre-order, out-of-stock, discontinued). The runtime catalogue (F-02.3) and checkout (F-04.x) read these states to drive PDP availability copy, ATC button behaviour, and dispatch SLAs (TC-18).

**Background / Context**

Premium furniture has a non-trivial inventory model (BRD §18.4): made-to-order configurations have production lead times, large items can be pre-ordered against incoming containers, and SKU stock is variant-specific (a sofa with three fabric grades and two leg finishes is six logical SKUs). Without an admin surface, every stock correction or lead-time bump is a code/DB ticket, and customers see stale availability copy on PDPs.

**User Story**

As an Ops admin, I want to view stock levels, set low-stock thresholds, and adjust lead times per configuration so that PDP availability and dispatch SLAs reflect reality without engineering involvement.

**Flow — Happy Path**

1. Admin opens `/admin/inventory` (SCR-A-10). Default view: list of SKUs with low-stock or recent-movement flags.
2. Filters: product, category, inventory state, low-stock only, lead-time bucket, last-movement window.
3. Admin clicks a SKU → opens **Inventory Detail**: current quantity-on-hand, allocated (in active checkout sessions / unfulfilled orders), available (`on_hand − allocated`), reorder point, lead-time days, inventory state.
4. Admin can: adjust stock (with mandatory reason: receipt, count correction, write-off, transfer); change inventory state; edit lead-time; edit low-stock threshold; bulk-adjust via CSV.
5. On stock change, `available` is recomputed; if available falls to or below threshold, an alert is generated for Ops dashboard (F-09.4) and the daily digest (F-07.4 Admin Alerts).
6. All write actions written to audit log with admin ID, timestamp, before/after diff.

**Flow — Edge Cases**

| Edge Case | Expected System Behaviour |
|---|---|
| Stock adjusted negative due to count correction | Allowed (with `count correction` reason). System logs negative balance, blocks new checkouts on that SKU until restored. |
| Inventory state set to `discontinued` while SKU is in active carts | Carts retain item until session expiry (TC-12) or checkout. PDP hides ATC and shows "Discontinued — view alternatives". |
| Lead-time increased while order is in-flight | In-flight orders unchanged. New PDP visits and new checkout sessions see the new lead-time. |
| Bulk CSV upload contains invalid SKU codes | Per-row validation report; valid rows committed, invalid rows skipped with error column. |
| Made-to-order SKU adjusted to `in_stock` | Allowed; supplier flag cleared; F-05.3 fulfilment SLA falls back to TC-18 from confirmation. |
| Concurrent admin edit (two admins same SKU) | Optimistic locking via `updated_at`. Second writer receives 409 with diff preview. |

**Business Rules**

| BR # | Rule |
|---|---|
| BR-01 | `available = on_hand − allocated`. `allocated` is the sum of (a) line items in non-expired checkout sessions and (b) confirmed-but-unfulfilled order line items. |
| BR-02 | Inventory states form a controlled set: `in_stock`, `made_to_order`, `pre_order`, `out_of_stock`, `discontinued`. State transitions are admin-driven; the runtime never auto-transitions to `discontinued`. |
| BR-03 | An `in_stock` SKU with `available ≤ 0` is auto-flagged `out_of_stock` for storefront purposes; admin must explicitly mark `discontinued` for permanent removal. |
| BR-04 | Lead-time is per configuration (variant), not per parent product. PDP displays the longest lead-time among selected variant options. |
| BR-05 | Stock adjustment requires a reason from the controlled vocabulary. |
| BR-06 | Low-stock alert generated when `available ≤ reorder_point` and previous reading was above. Alert is debounced — at most one per SKU per 24h. |
| BR-07 | All inventory writes are audited; bulk uploads recorded with file hash and row count. |
| BR-08 | Admin — Content cannot mutate inventory; can read product-page-relevant fields (state, lead-time copy) only via product mgmt (F-02.5). |

**Input Validation Rules**

| Field | Type | Required | Validation Rules | Error Message Shown |
|---|---|---|---|---|
| sku | String | Yes | Must exist; `[A-Z0-9-]+`; max 32 chars. | 'SKU not recognised.' |
| adjustment_qty | Integer | Yes | Non-zero; abs ≤ 100,000. | 'Adjustment must be a non-zero integer.' |
| reason | Enum | Yes | One of: `receipt`, `count_correction`, `write_off`, `transfer_in`, `transfer_out`, `return_restock`, `damage`. | 'Please select a reason.' |
| lead_time_days | Integer | Yes (for made-to-order) | 1–180. | 'Lead time must be 1–180 days.' |
| reorder_point | Integer | Yes | ≥ 0. | 'Reorder point cannot be negative.' |
| inventory_state | Enum | Yes | One of the 5 states above. | 'Invalid inventory state.' |
