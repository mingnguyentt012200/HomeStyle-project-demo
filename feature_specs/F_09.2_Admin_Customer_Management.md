## M09 — Admin Operations (Logistics & Commerce Configuration)

---

### F-09.2 Admin Customer Management

**Summary**

Ops admins (and Super Admin) view, search, and manage registered customer and trade-account records: see lifetime value, order history, addresses, marketing consent, and activate / deactivate accounts. Provides the customer 360° view used by support, fulfilment, and fraud workflows. Read-only fields are presented to Admin — Trade for B2B customers in their book.

**Background / Context**

BRD §8.4 lists "Customer management: view/search, order history, activate/deactivate, trade account status" as a Phase-1 Admin Panel capability. Without it, support agents have no GUI to assist callers, fraud cases cannot be locked out, and GDPR data-subject requests (F-08.2) cannot be triaged efficiently. F-09.2 is the operational front door to the Customer entity; F-08.2 is the destructive (right-to-erasure) endpoint.

**User Story**

As an Ops admin, I want to search and view customer records, see their order and return history, and activate or deactivate accounts so that I can resolve support cases and act on fraud or abuse reports without engineering involvement.

**Flow — Happy Path**

1. Admin opens `/admin/customers` (SCR-A-09 list view). Filters: status (active/deactivated), trade vs consumer, country, marketing-opt-in, date created, lifetime spend bucket.
2. Admin searches by email, name, customer ID, or order number. Search is server-side, case-insensitive.
3. Admin clicks a row → opens **Customer Detail** (SCR-A-09 detail). Tabs: **Overview**, **Orders**, **Returns**, **Addresses**, **Reviews**, **Consent & Privacy**, **Audit**.
4. **Overview**: profile, lifetime value, order count, AOV, last order date, trade status (if any), AM (if assigned), notes.
5. **Orders / Returns** tabs: paginated lists, deep-link into F-05.3 / F-06.2.
6. Admin can: edit notes, deactivate account (soft, with reason), reactivate, trigger password reset email (F-01.3), trigger data-export (F-08.2 portability), or open right-to-erasure flow (F-08.2 deletion).
7. All actions written to audit log with admin ID, timestamp, before/after diff.

**Flow — Edge Cases**

| Edge Case | Expected System Behaviour |
|---|---|
| Admin attempts to deactivate a Verified Trade Account | Block in Ops UI. Show: "Trade accounts must be deactivated by Admin — Trade." Deep-link to F-10.1. |
| Customer has open order(s) when deactivated | Allow deactivate; existing orders proceed normally. New logins blocked with localised "Account deactivated — contact support" message. |
| Customer has pending GDPR erasure request | Show banner; deactivation allowed but erasure timer (30 days, BR-012) continues regardless. |
| Search returns >500 results | Paginate (50/page) with server-side cursor. UI nudges admin to add filters. |
| Admin tries to view PII fields without scope | Fields masked (e.g. `j***n@***.com`). Reveal requires elevated re-auth (step-up MFA). |

**Business Rules**

| BR # | Rule |
|---|---|
| BR-01 | Customer records are never hard-deleted from this surface. Erasure is via F-08.2 only. |
| BR-02 | Deactivation revokes all sessions immediately (token revoke per F-01.5). |
| BR-03 | Deactivation reason is mandatory; selectable from a controlled vocabulary (Fraud, Abuse, Customer Request, Duplicate, Other). |
| BR-04 | Trade account lifecycle (approve/suspend/reactivate) is owned by Admin — Trade (F-10.1). Ops can view but not mutate trade status. |
| BR-05 | Email/phone PII reveal requires step-up MFA within last 5 minutes; logged. |
| BR-06 | Marketing-consent edits require a verified source (admin acting on customer request, recorded ticket ref). |
| BR-07 | All write operations are audited (BR-007 of master). |
| BR-08 | Lifetime value, order count and AOV are computed from confirmed orders only (excl. cancelled/refunded). |

**Input Validation Rules**

| Field | Type | Required | Validation Rules | Error Message Shown |
|---|---|---|---|---|
| q (search) | String | No | Trimmed; 2–80 chars. | 'Search term must be at least 2 characters.' |
| deactivation_reason | Enum | Yes (on deactivate) | Must be in vocabulary. | 'Please select a deactivation reason.' |
| deactivation_note | String | No | Max 500 chars. | 'Note exceeds 500 characters.' |
| admin_note | String | No | Max 2,000 chars. Markdown stripped. | 'Note exceeds 2,000 characters.' |
