### F-09.1 Admin Shipping Configuration

**Summary**

Ops admins (and Super Admin) configure shipping zones, flat-rate and weight-based shipping rates, free-shipping thresholds, and white-glove delivery partners. The runtime checkout calculator (F-04.4) reads the resulting `shipping_rates`, `shipping_zones`, and `whiteglove_partners` tables to compute customer-facing shipping cost and delivery options.

**Background / Context**

HomeStyle ships a heterogeneous mix of small accessories (parcel carriers), medium furniture (threshold/2-person delivery) and large furniture (LTL + white-glove) across US and EU markets (BRD §18.2). Rates change as carrier contracts are renegotiated, fuel surcharges shift, and new white-glove partners are onboarded for new ZIP/postal regions. Without an admin UI, every change is a code/DB ticket — unacceptable at premium-furniture scale. F-04.4 is the *consumer* of this data; F-09.1 is the *producer*.

**User Story**

As an Ops admin, I want to define shipping zones, weight-based rates, and white-glove partners so that checkout shows accurate shipping costs and eligibility without engineering involvement.

**Flow — Happy Path**

1. Admin opens `/admin/shipping` (SCR-A-11). Three tabs: **Zones**, **Rates**, **White-Glove Partners**.
2. **Zones tab**: admin creates a zone (e.g. "EU-DE-North"), assigns ISO country code + postal/ZIP prefixes, sets currency.
3. **Rates tab**: admin selects a zone and adds rate row(s): `delivery_method` (parcel | threshold | white-glove), `flat_rate`, `per_kg_rate`, `min_weight_kg`, `max_weight_kg`, `free_shipping_threshold` (optional), `eta_days_min`, `eta_days_max`.
4. **White-Glove Partners tab**: admin creates partner record — name, contact, service ZIP/postal prefixes, surcharge amount, capabilities (room-of-choice, packaging removal, assembly).
5. Admin clicks **Save**. System validates rate ranges do not overlap within the same zone+method, and white-glove ZIP coverage does not duplicate.
6. On save, configuration is versioned (audit log, F-13.x). New rates take effect for *new* checkout sessions only — in-flight sessions retain their locked rate per TC-14.

**Flow — Edge Cases**

| Edge Case | Expected System Behaviour |
|---|---|
| Admin creates overlapping weight ranges in the same zone+method | 422 returned. UI highlights overlapping rows. |
| Admin deletes a zone with active orders referencing it | Soft-delete only. Zone marked `archived` and excluded from new lookups; historical orders retain reference. |
| Admin deletes a white-glove partner with scheduled deliveries | Block delete. Show count of scheduled deliveries; require reassignment first. |
| Free-shipping threshold set to 0 or negative | 422: 'Threshold must be greater than zero, or leave blank to disable.' |
| Admin uploads partner ZIP coverage CSV with invalid postcodes | Per-row validation report; valid rows imported, invalid rows rejected with reasons. |
| Admin changes rate that affects in-flight checkout sessions | Locked sessions unaffected (TC-14, BR-04). Banner warns admin: "Currently 23 active checkout sessions on prior rate." |

**Business Rules**

| BR # | Rule |
|---|---|
| BR-01 | Shipping rate formula: `flat_rate + (item_weight_kg × per_kg_rate)`, summed across line items, with white-glove surcharge added once if any line item triggers white-glove (per F-04.4 BR-05). |
| BR-02 | Zones are uniquely identified by `(country_code, postal_prefix_pattern)`. A postal prefix may belong to only one zone per delivery method. |
| BR-03 | White-glove eligibility is a function of (a) at least one line item with `requires_whiteglove=true` (large furniture per BRD §18.2) AND (b) destination ZIP/postal matches an active white-glove partner coverage. If (a) without (b): UI offers "threshold delivery" or "Contact us" per BRD §18.2. |
| BR-04 | Rate changes are non-retroactive. Locked checkout sessions (TC-14) and confirmed orders use the rate captured at lock time. |
| BR-05 | Only Admin — Operations and Super Admin can write to shipping configuration. Admin — Content has no access. Admin — Trade has read-only access (needed to advise B2B customers). |
| BR-06 | Free-shipping threshold is per zone+method. When met, F-04.4 shows `$0.00 — Free Shipping (orders over $X)`. |
| BR-07 | All write operations are written to the audit log with admin ID, timestamp, before/after diff. |
| BR-08 | Soft-delete only for zones, rates, and partners. Hard-delete forbidden to preserve historical order traceability. |

### F-09.1 Extension — Carrier Validation & Green Delivery Routing

*Spec version: 3.0 · Added per BA case study: EU logistics expansion (Poland warehouse → DE/FR/NL markets)*

**Summary**

Extends Admin Shipping Configuration with two operational capabilities: (1) **Carrier Validation** — automatic carrier eligibility checking based on packed weight and dimensions, preventing manual mis-assignment; (2) **Green Delivery Routing** — ESG-driven carrier selection for EU urban orders to meet the company's 30% green-delivery KPI and qualify for EU urban tax rebates.

**Background / Context**

With the central warehouse in Poland serving Germany, France, and the Netherlands, warehouse staff were manually selecting carriers without constraint enforcement. The result: standard parcel carriers (DPD EU, max 31.5 kg per consignment) being assigned to heavy furniture (e.g. 120 kg solid oak wardrobes), causing carrier refusals at collection, 2-day delays, and Trustpilot 1-star reviews. Additionally, EU policy now requires 30% of orders delivered into major urban zones (Berlin, Paris, Amsterdam) to use certified green carriers (EV fleet or carbon-neutral certified) to qualify for tax rebates. Without system automation, compliance depends entirely on individual warehouse judgment.

---

**Carrier Profile Management**

Each carrier configured in the system carries a `carrier_profile` record that the auto-assignment engine reads at fulfilment time (F-05.6).

| Field | Type | Description |
|---|---|---|
| `carrier_code` | String | Unique identifier (e.g. `DPD-EU`, `DHL-EXPRESS-EU`, `KN-LTL-EU`) |
| `carrier_name` | String | Display name |
| `service_type` | Enum | `parcel` \| `ltl` \| `white_glove` |
| `max_weight_kg` | Decimal | Per-consignment weight ceiling (e.g. 31.5 for DPD EU parcel) |
| `max_length_cm` | Decimal | Longest single dimension allowed |
| `max_girth_cm` | Decimal | (2×H + 2×W) ceiling |
| `max_volume_m3` | Decimal | Volumetric ceiling |
| `supported_zones` | String[] | Zone codes this carrier services |
| `green_certified` | Boolean | Carrier holds EU carbon-neutral cert or operates EV fleet |
| `green_cert_reference` | String | Certificate number / scheme (e.g. "DHL GoGreen Plus ISO 14064") |
| `active` | Boolean | Soft-disable without deleting |

**Carrier Auto-Assignment Logic (Runtime — F-05.6 Pack-Verify Step)**

After pack-verify barcode scan, before label generation:

1. Retrieve packed consignment weight + dimensions from packing station.
2. Filter `carrier_profiles` to `supported_zones` matching destination zone and `active = true`.
3. Eliminate carriers where `consignment_weight > max_weight_kg` OR any single dimension exceeds `max_length_cm` OR girth exceeds `max_girth_cm`.
4. **Green Delivery Check:** If destination postal code falls within a `green_delivery_zone`, retrieve the rolling 30-day green-delivery ratio for that zone. If ratio < `green_kpi_target_pct`: rank `green_certified = true` carriers first. If ratio ≥ target: standard cost-optimised ranking applies.
5. Auto-select top-ranked eligible carrier. Present confirmation modal to warehouse operator (carrier name, service type, estimated cost, green badge if applicable).
6. Operator confirms or overrides (logged with mandatory reason code).
7. Label generated via carrier API. Tracking number written to `shipment_records`.

**Green Delivery Zone Configuration**

Admins configure EU urban postal-prefix lists for green routing. Initial scope: Berlin (`10xxx`–`14xxx`), Paris (`75xxx`), Amsterdam (`1xxx`).

| Field | Description |
|---|---|
| `zone_name` | Human label (e.g. "Berlin Urban") |
| `country_code` | ISO-3166-1 alpha-2 |
| `postal_prefixes` | Array of prefix patterns |
| `green_kpi_target_pct` | Rolling-30-day target % (default 30) |
| `rebate_scheme_ref` | EU tax rebate reference code (for finance reporting) |

**Green Delivery KPI Tracking**

- Rolling 30-day ratio computed per `green_delivery_zone` nightly.
- Surfaced on Admin Dashboard (F-09.4) as a KPI card: *Green Delivery Ratio* (target ≥ 30% per zone).
- Monthly CSV export for EU tax rebate submission (shipment ID, carrier, zone, green flag, dispatch date).

---

**Edge Cases (Carrier Validation & Green Delivery)**

| Edge Case | Expected System Behaviour |
|---|---|
| Packed weight exceeds ALL configured carriers for destination zone | Flag `CARRIER_UNRESOLVED`; block label generation; alert Ops Admin via F-07.4 within 60 s. Manual resolution required. |
| Carrier API unavailable at label generation | Retry ×3 at 30 s intervals. After 3 failures: alert Ops Admin; offer manual label entry fallback (Super Admin override). |
| Operator overrides auto-assigned carrier with an ineligible one | System warns: "Selected carrier exceeds weight/dimension limits. Confirm to proceed at own risk." Requires reason code; logged for monthly audit. |
| Green KPI already met for that zone this rolling period | Green carrier preference disabled; standard cost-ranking applies until ratio drops below target again. |
| New carrier added mid-month with `green_certified = true` | Immediately eligible; KPI ratio recalculated nightly to include newly assigned shipments. |
| Carrier updates max weight (contract renewal) | Admin updates `carrier_profile.max_weight_kg`; non-retroactive (already-dispatched shipments unaffected). |

---

**Additional Business Rules**

| BR # | Rule |
|---|---|
| F-09.1-BR-09 | Auto-assignment engine must reject any carrier whose `max_weight_kg` < packed consignment weight. Manual override requires explicit reason code and is audit-logged. |
| F-09.1-BR-10 | DPD EU parcel constrained to ≤ 31.5 kg per consignment. Furniture SKUs with `requires_ltl = true` or `packed_weight_kg > 31.5` must route to an LTL or white-glove carrier. |
| F-09.1-BR-11 | For shipments into a configured `green_delivery_zone`, system preferentially selects `green_certified = true` carriers when the rolling 30-day ratio for that zone is below `green_kpi_target_pct`. |
| F-09.1-BR-12 | Green delivery ratio is computed per zone on a rolling 30-day basis. Zones below target for two consecutive months trigger a Logistics Director alert. |
| F-09.1-BR-13 | Carrier profiles are soft-deleted only (`active = false`). Deactivation removes carrier from future auto-assignment but preserves historical shipment references. |
| F-09.1-BR-14 | If no eligible carrier is found after validation, label generation is blocked and shipment enters `CARRIER_UNRESOLVED`. Ops Admin manual resolution required; TC-17/TC-18 SLA clock continues. |
| F-09.1-BR-15 | `green_cert_reference` is mandatory for all carriers with `green_certified = true`. Missing reference blocks carrier activation. |
