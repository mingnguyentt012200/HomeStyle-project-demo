## M10 — Trade Portal Administration

---

### F-10.1 Trade Account Application Review (Approve / Reject / Suspend)

**Summary**

Trade Admins (and Super Admin) review B2B trade-account applications submitted via the public Trade Portal — verify business documents (registration, VAT/tax ID, professional credentials), approve or reject with reason, and manage the lifecycle of approved accounts (suspend, reactivate, deactivate). Approval triggers welcome email, default pricing-tier assignment, and account-manager assignment (F-10.3).

**Background / Context**

BRD §8.4 lists "Trade account management: review applications, approve/reject with reason" as Phase-1 scope; BRD §12.2 defines the Trade Application & Approval business process. F-10.1 is the operational manifestation of that process, owned exclusively by Admin — Trade per BR-005 of master.

**User Story**

As a Trade Admin, I want a queue of trade applications with all the supporting documents and decision history visible in one place so that I can review and decide each application within SLA without switching tools.

**Flow — Happy Path**

1. Admin opens `/admin/trade/applications` (SCR-TR-02 / SCR-TA-01..03). Default view: queue sorted oldest-first, status `submitted` and `under_review`.
2. Filters: status, country, business type (interior designer / architect / corporate buyer), document-completeness, age in queue.
3. Admin opens an application → **Application Detail** with tabs: **Business Info**, **Documents**, **Verification Notes**, **Decision History**, **Audit**.
4. Admin can run automated checks: VAT-number lookup (EU VIES service), business-registry lookup (where supported), email/domain reputation. Results stamped on record.
5. Admin marks status `under_review` (auto on first open if `submitted`).
6. Admin selects **Decision**: Approve, Reject, or Request More Info.
   - **Approve**: select default pricing tier (F-10.2), assign account manager (F-10.3), trigger approval email (F-07.3) with credentials/onboarding pack.
   - **Reject**: select reason from controlled vocabulary; optional internal note; trigger rejection email with reason summary (PII-safe template).
   - **Request More Info**: select missing document type(s); trigger automated email; status set to `info_requested`; SLA timer paused.
7. Approved account is provisioned: role `Verified Trade Account`, VAT-exempt eligibility flag (if EU VAT verified), trade-pricing visibility enabled.
8. Lifecycle actions on approved accounts: **Suspend** (e.g. payment fraud), **Reactivate**, **Deactivate** (terminal, with reason).
9. All actions audited.

**Flow — Edge Cases**

| Edge Case | Expected System Behaviour |
|---|---|
| VIES VAT lookup down | Manual override allowed with mandatory note. Re-check job queued for next 24h. |
| Application missing required document | Cannot Approve; UI blocks button with inline reason. Request More Info flow available. |
| Applicant resubmits after rejection | New application with reference to prior; admin sees prior decision in **Decision History**. |
| Approved account later flagged for fraud | Suspend → all sessions revoked (per F-09.2 BR-02); active orders pause for review. |
| Account manager deleted/transferred while assigned | Forced reassignment via F-10.3; orphaned accounts surface as alert. |
| Trade applicant exists as Registered Customer | On approval, role upgrades to Verified Trade Account; cart and order history preserved. |
| Concurrent decisions by two Trade Admins | Optimistic locking; second writer 409. Decision History shows lock-out trace. |

**Business Rules**

| BR # | Rule |
|---|---|
| BR-01 | Only Admin — Trade and Super Admin can change trade-status. (BR-005 of master.) |
| BR-02 | Decision (approve/reject) requires a reason from the controlled vocabulary; free-text note optional. |
| BR-03 | Rejection emails must not echo internal notes; only the customer-facing reason summary. |
| BR-04 | Approval triggers: role provisioning, VAT-exempt flag (if EU VAT validated, BR-004 of master), default pricing tier (F-10.2), AM assignment (F-10.3), welcome email (F-07.3). |
| BR-05 | Suspended account retains data but blocks all login (F-01.5 token revoke). |
| BR-06 | SLA: applications must reach a terminal decision within 5 business days of `submitted`. Breach surfaces on dashboard (F-09.4) and daily digest (F-07.4). |
| BR-07 | All decisions and lifecycle actions audited with admin ID, timestamp, before/after diff, and snapshot of supporting documents. |
| BR-08 | Documents are stored with at-rest encryption and 7-year retention (compliance default). Erasure honoured per F-08.2 if account is deactivated and the customer requests. |

**Status State Machine**

```
submitted ──▶ under_review ──▶ approved ──▶ active ──▶ suspended ──▶ active
                  │                                       │
                  ├──▶ info_requested ──▶ under_review    └──▶ deactivated (terminal)
                  │
                  └──▶ rejected (terminal — applicant may resubmit as new)
```

**Input Validation Rules**

| Field | Type | Required | Validation Rules | Error Message Shown |
|---|---|---|---|---|
| decision | Enum | Yes | `approve`, `reject`, `request_info` | 'Select a decision.' |
| reason_code | Enum | Yes (reject/request_info) | Controlled vocabulary | 'Select a reason.' |
| missing_docs | Array<Enum> | Yes (request_info) | At least 1 | 'Select at least one missing document.' |
| internal_note | String | No | Max 4,000 chars | 'Note exceeds 4,000 characters.' |
| default_tier_id | UUID | Yes (approve) | Must exist in F-10.2 tiers | 'Select a pricing tier.' |
| account_manager_id | UUID | Yes (approve) | Must be active AM in F-10.3 | 'Select an account manager.' |
