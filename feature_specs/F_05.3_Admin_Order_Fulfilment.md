## M05 — Order Management

---

### F-05.3 Admin Order Fulfilment (Process 11.4)

**Summary**

Admin — Operations confirms paid orders, monitors fulfilment SLAs, manages the carrier hand-off for both parcel and white-glove items, and handles fulfilment exceptions. This spec covers the **order-level workflow**. The downstream operational mechanics are split into companion specs:

- **F-05.5 Order Routing & Allocation** → Process 11.11 (auto-allocates lines to source warehouses, creates Hard Locks)
- **F-05.6 Warehouse Pick, Pack & Ship** → Process 11.12 (wave picking, pack-verify barcode match)
- **F-05.7 Made-to-Order Production Tracking** → Process 11.5 (supplier PO, production status updates)

**Background / Context**

Fulfilment SLA compliance directly affects three KPIs: Order Confirmation SLA (TC-17, ≥95%), Dispatch SLA (TC-18, ≥95%), and Cart Abandonment (drives repeat purchase). Without an admin surface, every confirmation and dispatch is a code/DB write, and SLA breaches go undetected.

**User Story**

As an Admin — Operations user, I want to triage the fulfilment queue, confirm orders within SLA, hand them to routing & pick/pack, and resolve exceptions — so that orders are fulfilled accurately within TC-17 and TC-18 and SLA-breach alerts (Process 11.10) stay close to zero.

---

**Flow — Happy Path (in-stock parcel order)**

1. Stripe `payment_intent.succeeded` webhook arrives. System creates order with status `Paid` and starts **TC-17 timer** (4 biz-hr confirmation SLA).
2. Order appears in `/admin/orders` queue. Admin filter: status, SLA badge (green/amber/red), delivery type, made-to-order flag, warehouse.
3. Admin opens order, validates configuration, address, lead time. Clicks **Confirm**. System:
   - Dispatches E-04 (Order Confirmed).
   - Hands order to **F-05.5 Order Routing** (Process 11.11). Routing engine has **TC-FUL-01: 30 seconds** to decide.
4. Routing creates allocation rows (Hard Locks per BR-INV-04) and N Shipment IDs (one per origin warehouse / carrier service class, BR-FUL-02).
5. **TC-18 timer** (48 biz-hr dispatch SLA) starts. Order status `Processing`.
6. **F-05.6 Pick, Pack & Ship** consumes the allocation (wave picking, pack-verify barcode match). On pack-verify success: Physical Stock decremented, Hard Lock released, status `Packed`.
7. Carrier scan-out updates status `Shipped`. E-05 dispatched with tracking link + white-glove window (where applicable).
8. Carrier delivery webhook (or admin manual mark) → status `Delivered`. **TC-19 returns window opens**; TC-24 schedules review prompt 7 days later.

**Flow — Happy Path (made-to-order item)**

1–3. As above through admin confirmation.
4. Routing engine (Process 11.11) routes MTO line to **F-05.7 Production** (Process 11.5). In-stock lines on the same order can ship separately by default (BR-FUL-06), or "ship complete" if the customer opted in at checkout.
5. Production places supplier PO. Customer order shows `In Production`.
6. Supplier marks `Ready for Dispatch`. **TC-18 dispatch SLA starts** (BR-FUL-08) — not from order date.
7. Standard routing+pick/pack flow resumes from step 6 above.

**Flow — Edge Cases**

| Edge Case | Expected System Behaviour |
|---|---|
| Admin attempts invalid status transition | F-05.4 Status Machine rejects with 422. |
| TC-17 SLA approaching 75% (3 biz-hr in) | Order flagged amber on Admin Ops dashboard (Process 11.10). |
| TC-17 SLA breached | Order flagged red 'Overdue'. Super Admin auto-emailed. KPI counter increments. |
| TC-18 SLA breached | Same as TC-17. Customer notification at admin discretion (not auto-emailed). |
| Routing engine cannot decide within TC-FUL-01 | Order queued for Super Admin manual allocation. Dashboard alert. |
| Customer requests cancellation BEFORE pick | Admin Ops can cancel; Hard Lock released; Stripe refund triggered; Physical Stock unchanged. |
| Customer requests cancellation AFTER pick | Blocked (BR-FUL-10). Requires Operations Admin override; converts to returns flow. |
| Made-to-order production delayed beyond confirmed lead time | Admin can update estimated dispatch date. Customer notification at admin discretion. Returns eligibility may shift per OD-08. |
| Mispick discovered at pack-verify | F-05.6 blocks pack. Order returns to F-05.6 wave. If repeats: Super Admin notified (BR-WHS-05). |
| Oversell detected at pick (Physical < Hard Locked) | Last-paid order auto-routed to On-Hold per BR-INV-09; E-OVERSELL apology+voucher email dispatched. |

**Business Rules**

| BR # | Maps to BRD | Rule |
|---|---|---|
| F-05.3-BR-01 | — | Status transitions are governed exclusively by F-05.4 Status Machine. F-05.3 issues transition requests; it does not encode the machine. |
| F-05.3-BR-02 | TC-17 / TC-18 | Confirmation SLA: 4 biz-hrs from `payment_intent.succeeded`. Dispatch SLA: 48 biz-hrs from confirmation (in-stock) or 48 biz-hrs from production-complete (made-to-order, BR-FUL-08). |
| F-05.3-BR-03 | BR-FUL-02 | A single Order ID can spawn multiple Shipment / Package IDs when items ship from different warehouses or carrier services. |
| F-05.3-BR-04 | BR-FUL-03 | Order Status is **derived** from child shipment statuses (e.g., `Partially Delivered` when one shipment is delivered and one is in transit). Admin cannot directly set a derived status. |
| F-05.3-BR-05 | BR-FUL-04 / 05 | Customer is charged a single combined shipping fee at checkout (BR-FUL-04). Internal cost allocation across shipments is admin-only reporting (BR-FUL-05). |
| F-05.3-BR-06 | BR-FUL-06 | Multi-line orders mixing in-stock and made-to-order ship the in-stock portion early by default. "Ship complete" is a customer opt-in at checkout. |
| F-05.3-BR-07 | BR-FUL-09 | White-glove items bypass wave picking; handled as single-order picks with carrier-appointment booking. |
| F-05.3-BR-08 | BR-FUL-10 | Customer-initiated cancellation is blocked after status `Picked`. Operations Admin override available; converts to return flow. |
| F-05.3-BR-09 | BC-40 | All admin order actions (confirm, hold, cancel, manual status set, override) audit-logged with admin ID, timestamp, before/after. |
| F-05.3-BR-10 | TC-29 | Outbound emails (E-03/04/05) use the transactional email retry policy (3× at 1 min, 1× at 15 min). Admin alerted if all retries fail. |
| F-05.3-BR-11 | Process 11.10 | SLA breaches surface via Process 11.10 monitoring. F-05.3 reads the badge; it does not own the timer. |

**Input Validation Rules**

| Field | Type | Required | Validation | Error Message |
|---|---|---|---|---|
| status (request) | Enum | Yes | Must be valid next status per F-05.4 | "Invalid status transition" |
| adminNotes | String | No | ≤ 2000 chars | "Notes must be ≤ 2000 characters" |
| cancellationReason | Enum | Yes (for cancel) | Controlled vocab: `customer_request`, `payment_dispute`, `oversell`, `fraud_review`, `supplier_failure`, `other` | "Please select a cancellation reason" |
| supplierRef | String | Yes (for MTO confirm) | Alphanumeric + hyphens, max 100 chars | "Supplier reference is required for made-to-order items" |
| estimatedProductionCompletionDate | ISO 8601 | Yes (for MTO confirm) | ≥ today | "Production completion date cannot be in the past" |
| shipCompleteOptIn | Boolean | No | — | — |

> Note: Tracking number, carrier, and packing slip generation are owned by **F-05.6 Pick, Pack & Ship**. F-05.3 reads these fields; it does not validate them on this endpoint.
