### F-01.6 Customer Account Profile Management

*This feature was implied by the registration and order flows but not explicitly specified in v2.0. Added in v2.1.*

**Summary**

Authenticated customers can view and update their account profile (name, email, phone), change their password, manage saved delivery addresses, view their trade account status, and close their account.

**Background / Context**

For premium design buyers — especially trade customers managing multiple project delivery addresses — robust account management is a baseline expectation. Without these APIs, the frontend has no endpoints to render or update the /account/profile and /account/addresses pages referenced throughout the spec.

**User Story**

As a registered customer, I want to update my personal details, manage my saved delivery addresses, and change my password from my account page so that my account reflects current information.

**Flow — Happy Path**

1. Customer navigates to /account/profile. Page loads via GET /api/v1/account/me.
2. 2. Customer updates first name, last name, or phone and clicks Save.
   3. 3. PATCH /api/v1/account/me updates only the supplied fields.
      4. 4. Customer navigates to /account/addresses. GET /api/v1/account/addresses returns saved addresses.
         5. 5. Customer adds a new address (POST), edits an existing one (PATCH), deletes one (DELETE), or sets a default (PATCH with isDefault: true).
            6. 6. Customer navigates to /account/security and changes password via PATCH /api/v1/account/password. All existing refresh tokens are revoked.
              
               7. **Flow — Edge Cases**
              
               8. | Edge Case | Expected System Behaviour |
               9. |---|---|
               10. | Customer attempts to change email | Email change requires re-verification: new email receives verification link. Old email remains active until new one is verified. |
               11. | Customer deletes their default address | If other addresses exist, first remaining address promoted to default. If no addresses remain, defaultAddressId set to null. |
               12. | Customer requests account closure | Initiates soft-delete flow per F-08.2 (Data Erasure). Account deactivated immediately; erasure scheduled per TC-31. |
               13. | Customer sets a non-existent address as default | 404 returned. |
              
               14. **Business Rules**
              
               15. | BR # | Rule |
               16. |---|---|
               17. | BR-01 | PATCH /account/me accepts any subset of updatable fields. Fields not supplied are unchanged. |
               18. | BR-02 | Email change requires re-verification of the new email before it replaces the old one. |
               19. | BR-03 | Password change via PATCH /account/password requires the customer to supply their current password for confirmation. |
               20. | BR-04 | Successful password change revokes all refresh tokens (same as F-01.3 reset). |
               21. | BR-05 | Maximum 10 saved delivery addresses per customer. Attempting to add an 11th returns 422. |
               22. | BR-06 | Exactly one address may be flagged as isDefault=true per customer. Setting a new default clears the previous flag atomically. |
              
               23. **Input Validation Rules**
              
               24. | Field | Type | Required | Validation Rules | Error Message Shown |
               25. |---|---|---|---|---|
               26. | firstName | String | No (PATCH) | Min 1, max 50 chars. | 'First name must be 50 characters or fewer' |
               27. | lastName | String | No (PATCH) | Same as firstName. | 'Last name must be 50 characters or fewer' |
               28. | phone | String | No | E.164 format (e.g. +12025551234). Max 20 chars. | 'Please enter a valid phone number' |
               29. | currentPassword | String | Yes (for PATCH /password) | Validated against stored bcrypt hash. | 'Current password is incorrect' |
               30. | newPassword | String | Yes (for PATCH /password) | Same rules as registration. Must differ from currentPassword. | 'New password must meet complexity requirements' |
               31. | address fields | Various | Yes (for POST /addresses) | line1 (max 100), line2 (optional, max 100), city (max 100), state/county (max 100), postalCode, country (ISO 3166-1 alpha-2). | Per-field: '[Field] is required' |

               44. **UI / Display Rules**

               46. |---|---|---|---|---|

               53. **API Contract**
              
               54. | Method | Endpoint | Description | Request Body / Params | Response |
               55. |---|---|---|---|---|
               56. | GET | /api/v1/account/me | Get profile | — | 200: { id, firstName, lastName, email, phone, tradeAccountStatus, createdAt } |
               57. | PATCH | /api/v1/account/me | Update profile | { firstName?, lastName?, phone? } | 200: { user } |
               58. | PATCH | /api/v1/account/email | Request email change | { newEmail, password } | 200: { message } / 409: email taken |
               59. | PATCH | /api/v1/account/password | Change password | { currentPassword, newPassword } | 200: { message } / 401: wrong current password |
               60. | GET | /api/v1/account/addresses | List addresses | — | 200: { addresses[] } |
               61. | POST | /api/v1/account/addresses | Add address | { line1, line2?, city, state, postalCode, country, isDefault? } | 201: { address } / 422: limit reached |
               62. | PATCH | /api/v1/account/addresses/:addressId | Edit address | { line1?, city?, postalCode?, isDefault?, … } | 200: { address } |
               63. | DELETE | /api/v1/account/addresses/:addressId | Delete address | — | 200: { message } |
               64. | POST | /api/v1/account/close | Close account | { password } | 200: { message } — schedules erasure |
