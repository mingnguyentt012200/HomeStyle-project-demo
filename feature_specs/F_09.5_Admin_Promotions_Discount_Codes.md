## M09 — Admin Operations (Logistics & Commerce Configuration)

### F-09.5 Admin Promotions & Discount Code Management

**Summary**

Content admins (and Super Admin) create, edit, schedule and retire discount codes — percentage and fixed-amount, customer-scope (all / consumer-only / trade-only), with usage limits, per-customer limits, expiry windows, and minimum-cart-value gates. The runtime application of codes by customers is handled by F-04.3; F-09.5 is the admin-side CRUD that produces the data F-04.3 consumes.

**Background / Context**

BRD §8.4 lists "Promotions: discount code CRUD — percentage and fixed, trade-only codes, usage limits, expiry" as a Phase-1 Admin Panel module. F-04.3 (already specced) covers customer-side code redemption and stacking rules but does not specify how codes are produced. Without this spec, marketing cannot launch campaigns without an engineering ticket.

**User Story**

As a Content admin, I want to create, schedule and retire discount codes with clear rules and usage caps so that I can run campaigns without engineering involvement and without risk of promotion abuse.

**Flow — Happy Path**

1. Admin opens `/admin/promotions` (Content role surface). List view: code, type, status (draft/scheduled/active/expired/retired), redemptions vs cap, valid window.
2. Admin clicks **Create Code**. Wizard:
   - Step 1 — Code: name (internal), code string (auto-generate or custom), display label.
   - Step 2 — Discount: type (percent / fixed), value, currency (for fixed only), maximum discount cap (optional, for percent).
   - Step 3 — Eligibility: customer scope (all / consumer / trade), minimum cart value, included/excluded products or categories, single-use vs multi-use, per-customer cap.
   - Step 4 — Schedule: start/end timestamps, timezone.
   - Step 5 — Review & save (status = `scheduled` if start in future, `active` if start ≤ now).
3. Admin can later: edit non-active codes freely; edit active codes with restrictions (cannot change `value` or `type`); retire any code (sets status `retired`, prevents new redemptions).
4. Bulk operations: CSV import for batched single-use codes (e.g. 1,000 unique codes for an influencer campaign).
5. All write actions audited.

**Flow — Edge Cases**

| Edge Case | Expected System Behaviour |
|---|---|
| Admin tries to set `trade-only` with no Trade customers in system | Allowed; warning shown. |
| Active code edited mid-campaign (changing value or type) | Block with 422; require admin to retire and create new. Cosmetic edits (display label, end date forward) allowed. |
| Code expiry passed but redemptions still in-flight (locked at checkout) | Locked sessions honour the prior code (TC-14). New attempts return F-04.3 error. |
| CSV upload of 10,000 unique codes | Streamed processing with job ID; progress endpoint. Per-row errors reported. |
| Two admins edit the same code concurrently | Optimistic locking via `updated_at`; second writer 409. |
| Code string collides with existing code (case-insensitive) | 422 with suggestion. |
| Stacking attempt against another active code | Stacking rules enforced by F-04.3 BR set; admin only declares whether *this* code is stackable. |

**Business Rules**

| BR # | Rule |
|---|---|
| BR-01 | Code strings are case-insensitive on input; stored uppercase; alphanumeric + hyphen, 3–32 chars. |
| BR-02 | Type and discount value are immutable once any redemption exists; admin must retire and create new. |
| BR-03 | Trade-only codes are invisible to Guest/Customer roles at the customer-facing endpoint (BR-003 of master). |
| BR-04 | Total-cap and per-customer-cap are enforced server-side at redemption (F-04.3); F-09.5 stores them as fields on the code record. |
| BR-05 | Minimum cart value is evaluated against subtotal **after trade discount** (if applicable) and **before** tax/shipping. |
| BR-06 | Status transitions: `draft → scheduled → active → expired` (auto on schedule) and `retired` (manual; from any state). Retired is terminal; reactivation requires creating a new code. |
| BR-07 | Discount stacking with trade pricing or another code is governed by F-04.3 BR set. The admin sets `stackable: bool` here; the runtime decides eligibility. |
| BR-08 | All write actions audited; CSV uploads recorded with file hash and row count. |

**Input Validation Rules**

| Field | Type | Required | Validation Rules | Error Message Shown |
|---|---|---|---|---|
| code | String | Yes | Unique (case-insensitive); `[A-Z0-9-]{3,32}`. | 'Code must be 3–32 alphanumeric characters.' |
| type | Enum | Yes | `percent` or `fixed`. | 'Select discount type.' |
| value | Decimal | Yes | percent: 1–100; fixed: > 0 with currency. | 'Value out of range.' |
| max_discount | Decimal | No (percent only) | > 0 if set. | 'Max discount must be greater than zero.' |
| customer_scope | Enum | Yes | `all`, `consumer`, `trade`. | 'Select customer scope.' |
| min_cart_value | Decimal | No | ≥ 0. | 'Minimum cart value cannot be negative.' |
| start_at / end_at | Timestamp | Yes | end_at > start_at. | 'End must be after start.' |
| total_cap / per_customer_cap | Integer | No | ≥ 1 if set. | 'Cap must be ≥ 1.' |
